# Регламент взаимодействия менеджеров сервисов

Версия регламента: `1.0`

Владелец продукта и главный менеджер: пользователь проекта.

Менеджер SaaS Core и документации: менеджер `market-signal-ai-bot`.

## Назначение

Этот регламент можно передать агентам-менеджерам `stock-signal-scanner` и `telegram_company_matcher_app` как постоянную рабочую инструкцию. Это не лицензия и не даёт права самостоятельно менять продуктовые, финансовые или security-решения.

## Источники истины

1. Межсервисный контракт: `market-signal-ai-bot/docs/integration-contract.md`.
2. Задачи и статусы версии: `market-signal-ai-bot/docs/versions/v0.1-task-board.md`.
3. Production gate: `market-signal-ai-bot/docs/production-checklist.md`.
4. Локальная реализация каждого сервиса и его тесты остаются ответственностью менеджера этого сервиса.

При расхождении документа, кода и устного задания работа останавливается на dev, а конфликт передаётся менеджеру SaaS Core. Production по спорному контракту запрещён.

## Полномочия

- Менеджер сервиса может распределять утверждённые задачи внутри своей команды, менять внутреннюю реализацию и добавлять тесты.
- Менеджер сервиса не может самостоятельно менять `contractVersion`, обязательные поля, стоимость квоты, ownership данных, HMAC-схему, fail-closed политику или публичные юридические формулировки.
- Изменение продукта, тарифа или контракта утверждает главный менеджер.
- Изменение security controls дополнительно проверяет Роман.
- Production deploy выполняется только через Макса после QA и security approval.

## Безопасная связь

- Никогда не передавать секреты, токены, private keys и полные API-ключи в чатах, GitHub, документации, отчётах или скриншотах.
- В отчётах указывать только имя секрета, key id и статус настройки.
- Значения передаются через Cloudflare Secrets или другое согласованное защищённое хранилище.
- Для каждого сервиса и окружения используется отдельный секрет.

## Порядок работы

1. Менеджер получает Task ID, цель, контракт, критерии приёмки и путь к файлу задания.
2. До изменений он проверяет текущий код и сообщает найденные расхождения.
3. Реализация и миграции сначала выполняются локально, затем в dev.
4. Менеджер запускает типизацию, unit/contract tests и негативные security tests.
5. Отчёт записывается в файл задания или task board и передаётся главному менеджеру.
6. Роман проверяет безопасность, Лена проводит интеграционную регрессию, Макс готовит deploy/rollback.
7. Production разрешается только после общего PASS и явного решения главного менеджера.

## Правила идентификаторов

- Бизнес-idempotency key сохраняется при безопасном retry.
- Transport `X-Request-Id` для HMAC является одноразовым и меняется при каждом новом HTTP retry.
- `requestId + ticker` определяет идемпотентность решения квоты в контракте `1.1`.
- Нельзя переиспользовать один transport request id с другим телом, query, endpoint или timestamp.
