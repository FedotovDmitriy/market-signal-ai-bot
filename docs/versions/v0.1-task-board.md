## Product Decision: Read-Only News Channels And Private Premium Ticker Requests

Decision date: `2026-07-09`

Confirmed product model:

- Telegram channels are read-only broadcast feeds by country and language.
- Users do not send ticker requests inside country/language channels.
- Personal ticker analysis requests are available only through private Telegram bot flow, website account, or API according to subscription plan.
- Premium users receive access to private ticker-analysis actions; other channel subscribers only read broadcast news/signals.
- `telegram_company_matcher_app` publishes matched news to country/language channels through Core delivery.
- `market-signal-ai-bot` owns user access, subscription, quota, route configuration, and Telegram delivery.
- `stock-signal-scanner` performs ticker analysis only after Core access/quota approval.

Scope reviewed:

Confirmed controls:

- Broadcast and private destinations are separate in Core. Matcher delivery accepts only structured `recipient.type`: `channel` with `routeId`, or `user` with `userId`; raw `chatId` in the matcher payload is rejected.
- Channel delivery resolves `telegram_chat_id` only from Core-managed `bot_routes`; matcher cannot directly choose a Telegram destination.
- User delivery resolves `telegram_user_id` from Core `users`; scanner/matcher payloads do not carry Telegram bot tokens or raw destination secrets.
- Private ticker requests from Telegram groups/channels are ignored before analysis request creation.
- Free/basic users, expired subscriptions, and exhausted quota are denied before private analysis request creation.
- Automated regression passed: `npm.cmd run check`; `npm.cmd test` with `63/63` tests.

Security blockers:

1. **P0 - Telegram webhook can fail open if `TELEGRAM_WEBHOOK_SECRET` is absent.** The code verifies `X-Telegram-Bot-Api-Secret-Token` only when the env secret exists. Production must require this secret and fail closed when missing; otherwise forged webhook traffic can reach private ticker handling.
2. **P0 - private Telegram ticker flow has no `update_id`/message replay ledger.** Replayed Telegram updates can enqueue duplicate private analysis requests. The flow must store Telegram `update_id` or a deterministic message id per bot/env and return idempotently before creating another `analysis_requests` row or spending quota.
3. **P0 - private ticker quota check is preflight-only.** The flow checks `quota_balances`, then inserts `analysis_requests`, but does not reserve/debit quota through the same Core-owned quota decision path used by scanner access. This leaves a race/quota-bypass risk until private Telegram requests are wired through the canonical quota ledger.
4. **P1 - sender/destination binding is implicit.** The code uses `from.id` as the identity and `chat.id` as the reply destination, but does not explicitly require private-chat `from.id == chat.id`. Add this invariant and a negative test so forged or malformed updates cannot act as one user while sending acknowledgements to another chat.
5. **P1 - live scanner execution is not integrated.** Core records `telegram_private_bot` requests as queued and replies with acknowledgement, but no live scanner execution/result delivery path is approved. Until scanner execution and private delivery are connected through Core-owned quota/idempotency, the flow is not production-ready.

Required fixes before Roman approval:

- Require `TELEGRAM_WEBHOOK_SECRET` outside local/test mode and add a missing-secret negative test.
- Add Telegram webhook idempotency for `update_id` before business logic.
- Route private ticker requests through the same Core-owned quota/reservation/receipt path as API/scanner access, or create an equivalent atomic private-request ledger.
- Enforce `private` chat plus `from.id == chat.id` for private ticker commands.
- Add tests for replayed update, forged `from.id`/`chat.id` mismatch, quota race/idempotent retry, and missing webhook secret.

### Проверка HMAC, cache TTL и quota replay - 2026-06-30

Итоговый security-статус: `Заблокировано`

Статус internal HMAC P1: `Закрыто`

Статус production cache/quota contract: `Заблокировано - scanner остаётся доверенным источником скидки`

Подтверждено:

- Internal endpoints работают по HMAC-only. `Authorization: Bearer <INTERNAL_API_SECRET>` не даёт доступ без корректной подписи.
- Обязательны `X-Key-Id`, уникальный `X-Request-Id`, timestamp и signature. Каноническая строка связывает timestamp, key id, request id, HTTP method, path, canonical query и SHA-256 digest сырого тела.
- HMAC старше 5 минут отклоняется. Replay ledger из миграции `0011_api_auth_and_internal_replay.sql` хранит `(key_id, request_id)` 10 минут; повтор подписи возвращает `409 internal_request_replayed`.
- Cache hit принимается только при `cacheStatus=hit`, непустом `cacheCreatedAt`, совпадении `cacheGenerationVersion` с `generationVersion`, возрасте не из будущего и не старше policy TTL.
- Границы cache TTL проверены: 59 минут - hit, 60 минут - hit, 61 минута - miss с полной стоимостью.
- `forceRefresh=true` всегда даёт `bypass` и полную стоимость нового анализа.
- Цена берётся из server-owned `quota_plan_policies`; half-unit хранится целыми credits, без floating-point записи в D1.
- Идемпотентность использует составной ключ `(requestId, ticker)`. Повтор идентичного payload возвращает `own_repeat`/`own_repeat_fundrep` с `chargeUnits=0` и не увеличивает баланс повторно; изменённый payload для того же ключа отклоняется.
- Миграция `0012_access_contract_v11_half_credits.sql` успешно применена к локальной D1.

Блокирующее замечание:

- **P1, ответственный: Яша / владелец scanner contract.** HMAC предотвращает подмену cache-полей внешним клиентом, но не доказывает существование cache entry: scanner сам владеет HMAC-ключом и сам задаёт `cacheStatus`, `cacheCreatedAt`, `cacheGenerationVersion` и `generationVersion`. Он может подписать согласованный свежий `hit` без реального cache entry и получить пониженное списание. Проверки TTL и generation consistency не устраняют этот trust gap.

Требуемое исправление:

- Сделать скидку зависимой от server-verifiable evidence: core-owned cache ledger/receipt, отдельный доверенный cache service с подписанным receipt, либо lookup по cache entry id, который scanner не может создать самостоятельно.
- Альтернатива требует явного управленческого решения считать scanner доверенным billing oracle и принять риск письменно; без такого решения production gate не снимается.
- Добавить Workers-runtime/D1 integration test конкурентных одинаковых запросов. Текущие unit tests подтверждают state machine и последовательный replay, но используют mocked D1.

Проверки:

- `npm run check` - passed.
- `npm test` - passed, 37 tests.
- Добавлены негативные тесты Bearer-обхода, просроченной HMAC, подмены cache metadata, TTL 59/60/61, refresh и повторного списания.

### Финальный security review Core / Scanner - 2026-07-01

Итог: `FAIL`

Отдельные статусы:

- `API token P0`: `Закрыто в коде, deployment заблокирован конфигурацией`.
- `Internal HMAC P1`: `Криптография закрыта, authorization scopes не закрыты`.

Подтверждённые контроли:

- `POST /api/analysis/requests` требует API key с server-side scope `analysis:write`; проверяются keyed HMAC hash, active/revoked state, expiry, active user, active subscription и server-owned quota.
- Rate limits атомарно считаются в D1 по API key, user и IP. Legacy SHA-256 API keys не принимаются и требуют rotation.
- Создание ключа использует `API_KEY_HASH_SECRET`, Web Crypto randomness и показывает raw token только в create/rotate response. List API возвращает только prefix.
- Rotation выполняется одной D1 batch: старый key отзывается и новый keyed-hash key создаётся атомарно; тест overlap/retirement для service HMAC keys пройден.
- Core-owned cache ledger из `0013_core_cache_receipts.sql` не доверяет scanner cache hints. Discount появляется только по committed `core_cache_entries` после full-price new/refresh decision.
- Receipt выдаётся только разрешённому full-price решению, действует 15 минут, связывает immutable request/ticker/user/report/generation/language, принимает один digest и использует server time для cache TTL.
- Повтор commit с тем же digest идемпотентен; другой digest или immutable mismatch отклоняется. Conditional transitions и UNIQUE constraints защищают concurrent commit.
- Canonical string Core и Scanner совпадает: `timestamp.keyId.transportRequestId.METHOD.pathname.canonicalQuery.sha256(rawBody)`.
- Негативные тесты подтвердили отказ при неверном secret и подмене timestamp, key ID, request ID, method, path, query или body digest.
- Clock skew ограничен +/-5 минутами; nonce `(keyId, requestId)` хранится 10 минут. Повтор возвращает `409`, Bearer downgrade отсутствует.
- HMAC signature проверяется через Web Crypto-derived fixed-length MAC и constant-time byte loop; прямое строковое сравнение подписи для допуска отсутствует.

Нужен keyring с secret + scopes и `403 internal_scope_required` до nonce/бизнес-логики.

Влияние: Scanner читает `cacheReceiptId`, но не вызывает `/api/internal/access/cache/commit`; одновременно он продолжает требовать старые `cacheClaim/cacheClaimSignature`, которых Core не возвращает. Cached/own-repeat flow между реальными сервисами несовместим и fail-closed.
3. **P0 - HMAC production/dev bindings Scanner отсутствуют.** Владелец: Илья/Макс, DevOps. Срок: после исправления scopes, до dev smoke. Secret inventory показывает `CORE_HMAC_SECRET=false` для Scanner dev и production; локальный/remote config не содержит `CORE_HMAC_KEY_ID` и `ACCESS_CHECK_URL`. Реальный Core вызов невозможен.
4. **P0 - Core production keyring не готов.** Владелец: Илья/Макс. Срок: до production approval. Dev имеет `INTERNAL_API_SECRETS_JSON`, production его не имеет и сохраняет legacy `INTERNAL_API_SECRET` без `INTERNAL_API_KEY_ID`, поэтому production internal auth fail-closed.

7. **P2 - API key expiry input не валидируется при create/rotate.** Владелец: Яша. Срок: до public API beta. Невалидная непустая дата даёт `Date.parse=NaN` и фактически не истекающий key. Требуется ISO date validation и запрет past/invalid expiry.
8. **P2 - constant-time helper рано отклоняет строки разной длины.** Владелец: Яша. Срок: до security-ready. Это раскрывает только публичную длину HMAC, не secret, но не полностью соответствует fixed-length-first правилу; следует декодировать ровно 32 bytes и использовать `crypto.subtle.verify`/fixed-length comparison.
